1. System Resource Usage

These statistics are derived by periodically polling the values (every 10 seconds) and then averaging them over the report interval.

A. Java Virtual Machine Statistics

Average Load column indicates CPU utilization and is reported using the System Load Average for the last minute. This is based on Java SE's OperatingSystemMXBean.getSystemLoadAverage() call. Average Heap and Average Non Heap columns indicate Memory Utilization for JVM heap and Non-heap Memory respectively. JVM heap is created at JVM startup and is used to allocate memory for all class instances and arrays. It is reclaimed by GC. JVM non-heap is used for storing per-class structure such as field and method data, static, native code and shard among all threads. This area is not subject to be garbage collected.

2. Composite (Rollup) Statistics (Top 10)

Composite level aggregate statistics indicate flow rate (throughput/transactions per second) and latency (in milliseconds) for the endpoints and internal backups.

A. Endpoint Statistics

Indicate flow rate (throughput) and latency for the endpoints. Composite (rollup) Endoint statistics are derived by aggregating individual endpoint statistics (shown in Section 3) by taking weighted average of latency values for all endpoints (inbounds and outbounds) for a composite. Example shown below, when there are two inbound endpoints (EP1 and EP2) of a composite (Composite1) with counts (total number of executions during the report interval) and latencies. Endpoint Composite Count Latency ------------------------------------------------------ EP1 Composite1 C1 L1 EP2 Composite1 C2 L2 Composite (Rollup) Inbound Latency = ((C1 * L1) + (C2 * L2)) / (C1 + C2) Latency: Defined as the round-trip time for the thread to return back at the measurement point. For inbound endpoints a timer is started as soon as client thread enters Binding/Adapter layer. The client thread then goes though Service Infrastructure to downstream components as defined in the flow. The timer is stopped when the thread returns back from Service Infrastructure to the measurement point. This time spent in round-trip represents inbound latency statistics for this endpoint for one execution. Similarly for outbound endpoints the timer is started just before the call to external service and stopped when the thread returns. For a given endpoint the latencies for all such executions in a report interval are aggregated and averaged by taking sum of execution times and dividing by the number of executions. Total column represents the total number of successful executions either during the report interval or cumulatively. Fault column indicates total number of failed executions either during the report interval or cumulatively - calls that resulted in exceptions being thrown from downstream component. TPS column represents transactions per second. This is calculated by dividing the total number of executions by the report interval.

B. Backups in Internal Queues

Composite (rollup) Backup statistics are derived by combining data for all internal queues (details in Section 4) for all processes in a composite. Added: The number (count) of messages added to the queue during the report interval. Removed: The number of messages removed from the queue during the report interval. Closed: The number of messages closed during the report interval. Active value represents the number of messages being processed since the server start till the end of report interval.

5. Longest running Business Process (Top 10)

This section lists the aggregate statistics for top Asynchronous and Synchronous business process instances based on execution time. The statistics also include the Start Count (number of instances started), Completed Count (number of instances completed) and Faulted Count (number of instances faulted) during the report interval.

A. Asynchronous Processes

Top longest running asynchronous business processes. Asynchronous Process instances can get dehydrated while waiting for callback to arrive. Once the callback arrives the instances are loaded back by engine for further execution. This process can repeat multiple times for processes with multiple callbacks. These De-hydration/Re-hydration counts are captured and shown for asynchronous process statistics table below. The total time the instances were dehydrated is also shown (in seconds) under De-hydration Time column. Finally, once a callback arrives there might be more than trivial time for loaded engine to load the instance back. This is indicated by Callback Processing Time.
